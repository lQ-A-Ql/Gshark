     GShark-Sentinel 代码审计报告与下一步计划

     Context

     用户需要对 GShark-Sentinel 项目进行全面审计，评估当前开发路线是否偏离核心需

     结论

     开发路线在功能覆盖上没有偏离——所有需求模块均已实现且完整度较高。但在"简化流

     ---
     二、功能缺陷清单

     P0 — 体验层核心缺陷（直接影响核心需求）

     1. 缺少新手引导和快速开始流程
     - 打开应用后只有一个空白工作区，新用户不知道从哪里开始
     - 没有"打开示例文件"、"快速分析向导"等入口
     - 14 个功能页面平铺在侧边栏，新手无法判断分析顺序
     - 建议：添加欢迎页/快速开始向导，提供分析流程引导（打开文件 → 浏览概览 →
     筛选可疑 → 深入分析）

     2. 缺少"一键分析"快速工作流
     - 打开 PCAP 后需要手动逐个进入各分析页面
     - 没有自动检测流量类型并推荐分析模块的能力
     - 建议：打开文件后自动生成"分析摘要"面板，展示：检测到的协议类型、可疑流量
     数量、推荐的分析入口

     3. 威胁狩猎规则库过于薄弱
     - 仅 5 条 YARA 规则（SQL注入/XSS/RCE/Webshell/凭证泄露）
     - 完全缺少工控协议异常规则（如 Modbus 非法功能码、S7 未授权写入）
     - 完全缺少车机协议异常规则（如 UDS Security Access 暴力破解、CAN 异常帧率）
     - 完全缺少 CTF 常见模式规则（如隐写标记、编码嵌套、常见 flag 格式变体）
     - 建议：按场景分类扩充规则库至 30+ 条

     P1 — 功能完善缺陷

     4. 插件示例严重不足
     - 仅 1 个 JS 示例（简单 flag 检测），无 Python 示例
     - 缺少复杂检测逻辑示例（多包关联、统计聚合、onFinish 使用）
     - 缺少工控/车机专项检测插件示例
     - 建议：提供 4-6 个覆盖不同场景的示例插件

     5. XCP 和 KWP2000 未实现详细解析
     - docs/ctf-vehicle-industrial-focus.md 明确列为高频考点
     - 当前仅有识别提示，无明细解析页面
     - 建议：在 backend/internal/tshark/vehicle_analysis.go 中添加 XCP/KWP2000
     字段提取，前端车机页添加对应面板

     6. DBC 不支持 multiplexing 和 ARXML
     - README 已声明此边界，但对车机分析完整性有影响
     - 建议：中期支持 multiplexed signals，ARXML 可作为长期目标

     7. 工控分析缺少文档中建议的增强功能
     - docs/ctf-vehicle-industrial-focus.md 建议的三项均未实现：
       - Modbus "可疑写目标"排序
       - IEC104/DNP3 控制指令专门面板
       - S7 数据块十六进制转字符串辅助视图
     - 建议：按文档建议逐步实现

     8. 流量图缺少时间轴交互
     - 无法按时间段过滤查看流量分布
     - 无法点击图表中的数据点跳转到对应数据包
     - 建议：添加时间范围选择器和图表点击跳转

     P2 — 工程与体验缺陷

     9. DBC 配置不持久化
     - 导入的 DBC 文件路径仅存内存，重启后丢失
     - 建议：持久化到 localStorage 或后端配置文件

     10. 媒体分析无在线预览
     - 只能导出媒体工件，无法在应用内预览音视频
     - 建议：对常见格式（H.264/AAC）提供内嵌播放器

     11. 前端测试覆盖不足
     - 仅 17 个前端测试用例，交互逻辑测试缺失
     - 车机分析、工控分析、USB 分析等复杂页面无交互测试
     - 建议：补充关键页面的交互测试

     12. 大文件处理风险
     - 流重组在内存中进行，超大 PCAP 可能 OOM
     - 流缓存 FIFO 淘汰策略可能丢失重要流
     - 建议：添加流重组内存上限告警，优化淘汰策略（LRU + 优先级）

     13. 审计日志仅保留 200 条
     - 长时间分析会话中日志会被截断
     - 建议：支持日志导出，或增加保留上限

     14. CAN 信号缺少波形可视化
     - 有信号时间线数值（样本数/最值），但无波形图
     - 建议：用折线图展示 DBC 信号随时间变化的波形

     三、已实现功能的代码级缺陷

     3.1 packetStore — 带过滤的分页全量扫描 O(N)

     backend/internal/engine/packet_store.go:182-203

     当 predicate != nil 时，Page() 方法对整个 JSONL 文件做全量顺序读取 + JSON
     反序列化，每次分页请求都是 O(N)。对于 10 万+ 包的
     PCAP，每次翻页都要从头扫描到 cursor 位置。

     // Page() with predicate — 每次都从头遍历
     err := s.Iterate(predicate, func(packet model.Packet) error {
         if total >= cursor && len(items) < limit {
             items = append(items, packet)
         }
         total++
         return nil
     })

     实际上当前代码路径中，带 filter 的分页走的是 filteredPacketIndex（tshark
     预过滤 frame ID），所以这个 predicate 分支主要是 fallback。但如果 tshark
     filter 执行失败，就会退化到这个 O(N) 路径。

     影响：大文件 + 过滤失败时，翻页卡顿严重。
     建议：对 predicate 分页结果做缓存，或在 fallback 路径中限制扫描范围。

     3.2 snapshotByID — 每次调用都深拷贝整个 map

     backend/internal/engine/packet_store.go:310-319

     func (s *packetStore) snapshotByID() (string, map[int64]int64) {
         s.mu.RLock()
         defer s.mu.RUnlock()
         byID := make(map[int64]int64, len(s.byID))
         for id, offset := range s.byID {
             byID[id] = offset
         }
         return s.path, byID
     }

     每次调用 PageByIDs 或 ExistingIDs 都会完整拷贝 byID map。对于 50 万包的
     PCAP，这是 50 万次 map 写入。snapshot() 也有同样的问题（拷贝 offsets
     slice）。

     影响：高频分页场景下产生大量临时内存分配和 GC 压力。
     建议：使用 COW（copy-on-write）策略，或在 Append 完成后冻结 snapshot
     避免重复拷贝。

     3.3 displayFilterCache 无上限、无淘汰

     backend/internal/engine/service.go:1368-1413

     s.displayFilterCache[filter] = index

     displayFilterCache 是一个 map[string]*filteredPacketIndex，每个不同的
     filter 表达式都会缓存一份完整的 frame ID 列表。没有大小限制，没有 LRU
     淘汰。用户如果尝试多个不同的过滤表达式，每个都会保留在内存中直到下次
     LoadPCAP。

     影响：内存持续增长，尤其是大文件 + 频繁切换过滤器时。
     建议：添加缓存上限（如 16 条），LRU 淘汰最旧的条目。

     3.4 streamCache FIFO 淘汰 — 线性扫描 O(N)

     backend/internal/engine/service.go:890-897

     func (s *Service) removeStreamCacheOrderLocked(key string) {
         for i, v := range s.streamCacheOrder {
             if v == key {
                 s.streamCacheOrder = append(s.streamCacheOrder[:i],
     s.streamCacheOrder[i+1:]...)
                 return
             }
         }
     }

     streamCacheOrder 是一个
     slice，每次更新已存在的缓存条目时需要线性扫描找到位置再删除。默认上限
     256，影响不大，但如果用户调高 GSHARK_STREAM_CACHE_LIMIT 到
     4096，每次缓存更新都是 O(4096) 扫描 + slice 移动。

     影响：高缓存上限时性能下降。
     建议：用 container/list + map 实现标准 LRU。

     3.5 LoadPCAP 中 cancel 字段的竞态风险

     backend/internal/engine/service.go:109-110

     currentRunID := atomic.AddInt64(&s.runID, 1)
     runCtx, cancel := context.WithCancel(ctx)
     s.cancel = cancel  // 直接赋值，无锁保护

     s.cancel 的写入没有锁保护，而 StopStreaming() 读取 s.cancel 也没有锁：

     func (s *Service) StopStreaming() {
         if s.cancel != nil {
             s.cancel()
             s.cancel = nil
         }
     }

     如果两个 goroutine 同时调用 LoadPCAP 和 StopStreaming，存在竞态条件。

     影响：并发调用时可能 panic 或取消错误的 context。
     建议：用 s.mu 保护 s.cancel 的读写，或使用 atomic.Value。

     3.6 SetHuntingRuntimeConfig 直接操作环境变量

     backend/internal/engine/service.go:545-575

     _ = os.Setenv("GSHARK_YARA_ENABLED", "false")
     _ = os.Setenv("GSHARK_YARA_BIN", strings.TrimSpace(cfg.YaraBin))

     YARA 配置通过 os.Setenv / os.Unsetenv
     存储。环境变量是进程全局的，不是线程安全的（Go 的 os.Setenv
     本身有锁，但多个配置项的设置不是原子的）。更重要的是，这种设计意味着配置不
     持久化，重启后丢失。

     影响：配置不持久化；多个配置项的更新不是原子操作。
     建议：将 YARA 配置存储在 Service 结构体字段中（类似 tlsConf），用 s.mu
     保护。

     3.7 Objects() 方法的锁嵌套风险

     backend/internal/engine/service.go:597-670

     func (s *Service) Objects() []model.ObjectFile {
         s.objMu.Lock()        // 获取 objMu
         defer s.objMu.Unlock()
         // ...
         s.mu.RLock()          // 在持有 objMu 的情况下获取 mu
         pcapPath := s.pcap
         s.mu.RUnlock()
         // ...
         packetByObjectName := s.packetObjectNameIndex()  // 内部调用 
     packetStore.Iterate

     Objects() 持有 objMu 的同时获取 s.mu.RLock()，而 LoadPCAP() 中先获取
     s.objMu.Lock() 再获取 s.mu.Lock()。锁获取顺序一致（objMu →
     mu），所以不会死锁，但 packetObjectNameIndex() 会遍历整个
     packetStore，在持有 objMu 期间阻塞其他需要 objMu 的操作。

     影响：大文件时 Objects() 调用会长时间持有 objMu，阻塞并发请求。
     建议：先释放 objMu，完成耗时操作后再重新获取。

     3.8 HTTP 请求判断逻辑的误判风险

     backend/internal/engine/analysis.go:348

     if strings.HasPrefix(infoUpper, "1") || strings.HasPrefix(infoUpper, "2")
     ||
        strings.HasPrefix(infoUpper, "3") || strings.HasPrefix(infoUpper, "4")
     ||
        strings.HasPrefix(infoUpper, "5") {

     isHTTPRequestPacket 中，如果 Info 以数字 1-5 开头就认为可能是 HTTP
     响应。但很多非 HTTP 包的 Info 也以数字开头（如 DNS 查询 ID、TCP
     序列号等），这会导致误判。

     影响：非 HTTP 包可能被错误归类为 HTTP 响应，影响流重组准确性。

     backend/internal/engine/packet_store.go:321-342

     3.11 Python 插件进程泄漏风险

     backend/internal/plugin/runtime.go:436-483

     func (s *pythonPacketSession) Close() ([]model.ThreatHit, error) {
         if s.stdin != nil {
             _ = s.stdin.Close()
         }
         waitDone := make(chan error, 1)
         go func() {
             waitDone <- s.cmd.Wait()
         }()
         // ...
         case <-time.After(s.timeout):
             if s.cancel != nil {
                 s.cancel()
             }
             if s.cmd.Process != nil {
                 _ = s.cmd.Process.Kill()
             }

     如果 ProcessBatch 中 writeErr 发生后，Close() 仍然会尝试关闭 stdin
     并等待进程退出。但如果 Python 进程已经因为 stdin 关闭而退出，s.cmd.Wait()
     可能已经返回。更关键的是，如果 Close() 从未被调用（比如 ThreatHunt 中途
     panic），Python 进程会成为孤儿进程。

     影响：异常路径下可能泄漏 Python 子进程。
     建议：在 pythonPacketSession 中添加 finalizer 或确保 Close()
     在所有路径上被调用。

     3.12 SSE 事件通道溢出静默丢弃

     backend/internal/transport/http_server.go:877-885

     func (s *Server) broadcast(ev event) {
         s.mu.Lock()
         defer s.mu.Unlock()
         for ch := range s.clients {
             select {
             case ch <- ev:
             default:  // 通道满时静默丢弃
             }
         }
     }

     SSE 事件通道容量为 256。如果前端处理慢或网络延迟，事件会被静默丢弃。对于
     packet 事件这可能没问题（前端会轮询），但对于 status 和 error
     事件，丢弃意味着前端可能错过关键状态更新（如"解析完成"、"解析失败"）。

     影响：高负载下前端可能错过关键状态事件，导致 UI 状态不一致。
     建议：对 status/error
     事件使用更大的缓冲或保证投递；或在前端增加状态轮询兜底。

     3.13 前端 SentinelContext 状态膨胀

     frontend/src/app/state/SentinelContext.tsx

     整个应用状态集中在一个 Context 中（1300+ 行），包含 30+ 个 state 变量和 20+
      个 ref。任何状态变化都会触发所有消费者重渲染。虽然使用了 useMemo 包装
     value，但 value 对象本身在任何 state
     变化时都会重新创建（因为依赖数组包含所有 state）。

     影响：频繁的数据包更新（如预加载期间）会导致不必要的全局重渲染。
     建议：将 Context 拆分为多个独立的 Provider（如
     PacketContext、StreamContext、PluginContext），或使用 use-context-selector
     等库实现选择性订阅。

     3.14 前端预加载轮询的忙等待

     frontend/src/app/state/SentinelContext.tsx:990-1010

     while (Date.now() < waitDeadline) {
         // ... loadPacketPage ...
         if (parseFinishedRef.current) break;

     预加载使用 120ms 间隔的轮询循环，最长 120 秒。这意味着在预加载期间，前端每

     建议：利用已有的 SSE 事件机制，在后端解析完成时通过 SSE
