# 车机与工控分析合并摘要

## 一、合并范围

本文件按“车机 / 工控 / 旧工具页安全分析”方向合并根目录车机分析方案、零基础说明、CTF 车机工控关注点，以及前端旧工具页风格对齐轮次中的相关内容。

## 二、能力定位

- 车机分析面向 CAN、J1939、DoIP、UDS、DBC/信号映射和诊断安全链路。
- 工控分析面向 Modbus、控制指令、可疑写操作、规则命中和事务审计。
- 两类页面都应强调“协议字段 + 安全解释 + 可定位证据”的组合，而不是只展示统计图。

## 三、车机方向现状

### 1. 已实现方向

- CAN / J1939 / DoIP / UDS 协议识别与基础字段展示。
- 关键字段提取、聚合统计和安全审计建议输出。
- 车机页面已按 MISC 风格进行浅色卡片、指标、桶图和列表对齐。
- DBC 相关说明已形成后续路线：从 ID 统计升级到信号语义。

### 2. 待扩展方向

- OBD-II PID 级解析。
- CAN payload 语义解码和 DBC/ARXML 映射。
- UDS 请求-响应配对、耗时统计和负响应解释。
- DoIP + UDS 组合事务视图。
- 刷写流程、鉴权、安全访问和例程控制完整链路重建。

## 四、工控方向现状

### 1. 已实现方向

- 工控页面已完成基础风格对齐，风险标签、事务类型和提示块开始使用共享 `AnalysisBadge` / `AnalysisCallout`。
- Modbus 事务、规则命中、可疑写操作和控制指令已具备结构化展示入口。
- 页面视觉从旧工具页样式迁移到 MISC 准线，降低与新工具箱的割裂感。

### 2. 待扩展方向

- Modbus 功能码解释仍可继续细化，尤其是写线圈、写寄存器、多寄存器写入和异常响应。
- 工控规则需要继续补充误报抑制、资产上下文和操作风险解释。
- 复杂表格仍是页面内手写结构，适合迁移到增强后的 `AnalysisDataTable`。
- 缺少“控制行为时间线”，后续可以按设备、功能码、寄存器和写入值组织。

## 五、共性设计缺陷

1. 车机和工控都需要从“统计页面”继续升级为“调查工作台”，强化证据定位和过滤联动。
2. DBC/ARXML、UDS 配对、Modbus 语义解释属于协议深水区，需要真实样本和规则库共同推进。
3. 页面样式已对齐，但复杂表格、详情面板和风险说明仍存在本地实现重复。
4. 车机/工控的风险等级应该统一到共享 badge tone，不应继续散落字符串 class。
5. 当前文档分散在根目录方案、零基础说明、关注点和前端轮次报告中，本文件作为方向入口，但不替代原始材料。

## 六、下一步建议

1. 优先补 UDS 请求-响应配对和负响应解释，使车机页面具备诊断链路可读性。
2. 为 DBC 映射设计轻量导入与字段展示，不急于实现完整编辑器。
3. 为 Modbus 写操作建立风险时间线，突出“谁在什么时候写了什么寄存器”。
4. 继续迁移车机/工控复杂表格到共享数据表组件。
5. 将车机/工控高风险事件接入证据定位、过滤动作和报告导出。
